# Стратегия версионирования и GitVersion

## Назначение

Этот документ объясняет стратегию семантического версионирования, логику модели ветвления и механизмы автоматического вычисления версий, используемые в проектах Digital Foundation.

## Целевая аудитория

Разработчики, DevOps инженеры, релиз-менеджеры и QA инженеры, которым необходимо понимать, как вычисляются номера версий и как CI/CD пайплайны генерируют теги версий.

## Текущее состояние

Платформа использует **GitVersion v6.x** для автоматического и динамического вычисления семантических версий (SemVer) на основе нашей модели ветвления GitFlow и истории коммитов. Такой подход полностью исключает необходимость ручного повышения версий в коде или конфигурационных файлах.

## Ключевые детали

### 1. Модель ветвления (GitFlow)

Наше вычисление версий тесно интегрировано с моделью ветвления GitFlow. GitVersion понимает семантику различных типов веток и автоматически применяет соответствующие инкременты версий и предрелизные теги:

- **`main`**: Состояние, готовое к продакшену. Коммиты в `main` представляют стабильные релизы. По умолчанию слияние ветки `release` или `feature` в `main` вычисляет следующую стабильную версию (например, `1.1.1` или `1.2.0`).
- **`develop`**: Интеграционная ветка для следующего релиза. Версии здесь включают предрелизный тег `alpha`, указывающий на предстоящую версию (например, `1.2.0-alpha.2`).
- **`feature/*`**: Фиче-ветки наследуют версию `develop`, но добавляют название фичи как предрелизный тег для изоляции сборки (например, `1.2.0-my-feature.1`).
- **`release/*`**: Ветки стабилизации для подготовки к продакшену. Версии здесь используют предрелизный тег `beta` или `rc` (например, `1.2.0-beta.1`).
- **`hotfix/*`**: Экстренные исправления, нацеленные напрямую на `main`. Они повышают patch-версию (например, `1.1.2-beta.1`).

**Настраиваемые предрелизные теги (суффиксы)**
По умолчанию GitVersion использует суффикс `alpha` для ветки `develop` и `beta` для веток `release/*`. Однако они полностью настраиваются через файл `GitVersion.yml`. В индустрии популярны два подхода к именованию:
1. **По умолчанию**: `alpha` (для интеграции) и `beta` (для стабилизации релиза).
2. **Альтернативный**: `dev` (для интеграции, например `1.2.0-dev.1`) и `rc` (Release Candidate, например `1.2.0-rc.1`).

### 2. Общепринятые коммиты (Conventional Commits)

Хотя структура GitFlow сама по себе диктует общий поток версий, **Conventional Commits** могут использоваться для явного принудительного инкремента конкретной версии внутри ветки:

- `fix:` / `bugfix:` → Принудительный инкремент **Patch** (например, `1.1.0` → `1.1.1`).
- `feat:` / `feature:` → Принудительный инкремент **Minor** (например, `1.1.1` → `1.2.0`).
- **Major инкременты (Ломающие изменения)**: Вместо громоздкого `BREAKING CHANGE:` в подвале коммита, стандартным и самым лаконичным способом является добавление восклицательного знака `!` перед двоеточием (например, `feat!: rewrite API` или `fix(core)!: change behavior`). GitVersion также поддерживает явные теги в теле коммита, например `+semver: major`.
- **Остальные префиксы**: Такие префиксы как `docs:`, `chore:`, `refactor:`, `style:` и `test:` **не вызывают инкремента версии** по умолчанию, так как считаются незначительными для функционала релиза.

*Примечание: В GitVersion v6, если вы сливаете фиче-ветку в `develop` без префикса `feat:`, система все равно корректно повышает минорную версию, потому что слияние в `develop` по своей сути сигнализирует о непрерывном прогрессе минорной версии.*

### 3. Интеграция с CI/CD

GitVersion v6 не зависит от платформы. Мы поддерживаем его выполнение в различных инструментах CI/CD. Песочница с полностью рабочими примерами доступна в репозитории [levin-dmytro/gitversion](https://github.com/levin-dmytro/gitversion).

#### AWS CodeBuild
В стандартных окружениях CodeBuild (`aws/codebuild/standard:7.0`) предварительно установлен .NET 6.0. Самая чистая интеграция — через глобальную утилиту .NET:
```yaml
phases:
  install:
    commands:
      - dotnet tool install --global GitVersion.Tool --version 6.0.2
      - export PATH="$PATH:/root/.dotnet/tools"
  pre_build:
    commands:
      - IMAGE_TAG=$(dotnet-gitversion /output json /showvariable SemVer)
```

#### GitHub Actions
Используйте официальный GitTools Action:
```yaml
- uses: actions/checkout@v4
  with:
    fetch-depth: 0 # Обязательно для GitVersion

- uses: gittools/actions/gitversion/setup@v3.0.0
  with:
    versionSpec: '6.0.0'

- id: gitversion
  uses: gittools/actions/gitversion/execute@v3.0.0
  with:
    useConfigFile: true
```

#### GitLab CI
Используйте официальный Docker-образ для вычисления версии:
```yaml
determine_version:
  image:
    name: gittools/gitversion:6.0.2-debian.11-6.0
    entrypoint: ['']
  script:
    - VERSION=$(/tools/dotnet-gitversion /output json /showvariable SemVer)
```

#### Bitbucket Pipelines
Используйте официальный Docker-образ и убедитесь в полной глубине клонирования (clone depth: full):
```yaml
clone:
  depth: full # Обязательно для GitVersion
pipelines:
  default:
    - step:
        image: gittools/gitversion:6.0.2-debian.11-6.0
        script:
          - VERSION=$(/tools/dotnet-gitversion /output json /showvariable SemVer)
```
